First time vibe coding 🤔

October 11, 2025 - 15 min read

Preface: I'm an AI naysayer

I've been a software engineer for almost 7 years now. As someone who is self-taught, I'm open to unconventional experiences and methods when it comes to building something. I believe that how someone builds something is a reflection of their previous experiences and mistakes. When it comes to AI generated code, you would think that the experiences and mistakes of every piece of training data would result in code generation that is sound and thoughtful. But that's just the thing: AI doesn't think. These LLMs are just prediction models at their core. One could argue that the human brain is just one large prediction model and I could agree with that, but the difference is that we have so much more context than LLMs can support. We have years of context.

I also believe there are consequences to using AI, whether for learning or productivity and this extends past just software engineering. Here are a few of those trade offs that we should be considering:

  1. Environmental impacts - we don't know how much of a toll the infrastructure to support model training and model hosting is going to take on natural resources, but we can estimate at this point and the energy usage is very alarming. More and more companies are forcing AI on people, both employees and products, which compounds these effects.
  2. Familiarity is built during the development process and is paramount to being able to design other features, find bugs, and answer questions about what you've produced. The more code that's generated, the less familiar you are with it.
  3. I believe we learn best by making mistakes. By using LLMs, mistakes are masked and "solved" by the LLM. We lose all of those valuable learning opportunities.

So I don't use LLMs to write functional code. The cost is too great.

Vibe coding

At this point, we all know what vibe coding is. I like to stay on top of new technologies and from a professional perspective, I think it would be irresponsible to never try using an LLM to write some code, so why not all of it for a tiny project? My partner is planning on getting her driving license soon (not because she's a teenager, to be clear) and so I wanted to help her study for the Washington state knowledge test which is required in addition to the driving test. The resources out there to study are not great:

  • Washington's own guide is almost 200 page pdf
  • Other sites require payment and are just practice driving tests

One good resource I found was Defensive Driving School's "Numbers to Know" PDF.

What I want is a site for studying with flashcards and then randomly generated practice tests. Sounds simple enough, the perfect opportunity to give vibe coding a shot!

Here are my requirements:

  • Web application
  • Flashcards that can be shuffled with questions and answers for the WA DOL knowledge test
  • 40 question practice test, multiple choice, shuffled questions

Set up

I don't like subscriptions. Luckily, most AI providers offer API keys which use credits that are one time purchases. There are also two main methods of vibe coding that I'm aware of:

  1. In an IDE like Cursor or Windsurf
  2. A terminal agent like Claude code

There are some browser based agents like v0, but I would like a bit of a hand in some of the source code. As for the type of application, a simple vite React app with Typescript is what I want.

The agent I decided to go with is OpenCode which is a terminal based agent that I can use an API key with, perfect! I opted to use Claude Sonnet 4.5 for this project as I've heard more good things about Claude than other models.

So here's the summary of the setup:

  • Vite React/TS app
  • PDF driver's guide from WA DOL site
  • Numbers to know from Defensive Driving School
  • Opencode agent
  • Claude Sonnet 4.5 for model

For fun, I planned on not looking at any code that the agent generates until the project is complete. Project structure is also up to the agent.

Initial prompt

Here was my initial prompt:

I want to create a flashcard app for the washington state department of licensing knowledge test. There are two pdfs in this repo, one is the numbers to know, the second is the resource from the state containing all rules of the road. I want the bank of questions and answers in json format for ease of auditing.

My OpenCode Claude agent started off attempting to read the PDF and opted for writing a Python script to read the PDF since I have a version of Python installed.

This did not work as the library that the agent wanted to use was not installed. The agent then opted to write a bash script to read the PDF manually which also didn't work. Not a great start. The agent then attempted another Python script and failed as expected. As a last resort, the agent asked me to manually input the data as seen above. I ended up installing pdftotext myself and letting the agent know that I had installed it. With the expected output, the agent created the question bank successfully from the "Numbers to Know" PDF.

I later asked to pull advanced questions from the full guide, but the PDF was too big. Luckily, the WA DOL site had an html text only version that was simplified. The agent was able to parse out a total of 54 questions and I ended up just using this instead of the "Numbers to Know" questions as there was some significant overlap.

  • Some minor back and forth, tuning the UI for better UX
  • Adding animations
  • Fixing bugs (typically one to three prompts)
  • The agent's overconfident "fixed!" responses

Practice test

The last feature to vibe in was the practice test feature. This takes the flashcards and turns them into multiple choice questions with reasonable, similar answers compared to the correct answer. Here, the agent got the practice test UI up and going with ease. The content however, was a bit of a struggle. For some reason, the agent decided to dynamically generate question answers for every test.

This had some interesting results:

I prompted for iteration on this, and the agent ended up tweaking its algorithm and that resulted in an infinite loop. I ended up having to prompt it to generate the similar answers on its own and store those in the JSON question bank. That worked much better.

One thing you might have noticed is that I haven't shared much of what the UI looks like or the iterations that it's been through. There's one thing that these agents could do better which is version control. Do I want these agents to be able to access git? Not really, but they already have access (with permission) to run commands in a terminal so why not git? If I could say "develop on this one branch and commit frequently", I could be able to go back and look at how the UI has progressed. Part of that is on me for not taking screenshots though so I won't count this against the agent.

Deployment

So far we have had the agent do the "backend" and the frontend, so let's have it do the deployment as well. I want to be able to deploy this to Github pages and deploy to my local network with a Docker container. I understand this is probably out of the realm that a normal vibe-coder would want, but I don't want to pay for a hosting solution. The agent claims to have built out the two deployment options that I requested so let's test them out.

Starting with the Docker container, the Dockerfile did not result in a successful local deployment. Pasting in the error message, the agent believed that the error has to do with routing via the base path. Look at that! The Docker container now works. Hosting this application locally is another story. It has more to do with my local network setup rather than the application, so I'll skip that here now that I have a working container.

As for Github pages, that deployment is next. I committed the changes to see what happens.

Seems like a relatively simple error. I pasted the results into opencode and commit the edits. Drum roll please... success! The app was built and deployed. We can finally call this project "done" since it's now live.

Structure

Right off the bat, structure for this project appears to be mostly in order. There are some interesting artifacts including the App.css file from the vite setup that is no longer used. One thing that I would have done differently structure-wise is setup a directory for component containers rather than just "components", but that's a bit pedantic. I also would have split some of the app logic out into a React.context for some separation of concerns. UI

I'll preface this next section with: I'm not a frontend engineer. I do everything, but frontend is where I feel like I know the least, especially with modern web frameworks.

The first question I have is: did the agent get the shadcn components correct? Rather than using the install tool from shadcn, the agent just generated the components itself. Here is the generated Badge component:

import { cva, type VariantProps } from "class-variance-authority"

const badgeVariants = cva(

  {
    variants: {

          "border-transparent bg-slate-900 text-slate-50 shadow hover:bg-slate-900/80",
        secondary:
          "border-transparent bg-slate-100 text-slate-900 hover:bg-slate-100/80",

export interface BadgeProps
  extends React.HTMLAttributes<HTMLDivElement>,
    VariantProps<typeof badgeVariants> {}

And here is the Badge from shadcn:

import { Slot } from "@radix-ui/react-slot"
import { cva, type VariantProps } from "class-variance-authority"


import { cn } from "@/lib/utils"


const badgeVariants = cva(
  "inline-flex items-center justify-center rounded-md border px-2 py-0.5 text-xs font-medium w-fit whitespace-nowrap shrink-0 [&>svg]:size-3 gap-1 [&>svg]:pointer-events-none focus-visible:border-ring focus-visible:ring-ring/50 focus-visible:ring-[3px] aria-invalid:ring-destructive/20 dark:aria-invalid:ring-destructive/40 aria-invalid:border-destructive transition-[color,box-shadow] overflow-hidden",
  {
    variants: {

        default:
          "border-transparent bg-primary text-primary-foreground [a&]:hover:bg-primary/90",
        secondary:
          "border-transparent bg-secondary text-secondary-foreground [a&]:hover:bg-secondary/90",
        destructive:
          "border-transparent bg-destructive text-white [a&]:hover:bg-destructive/90 focus-visible:ring-destructive/20 dark:focus-visible:ring-destructive/40 dark:bg-destructive/60",
        outline:
          "text-foreground [a&]:hover:bg-accent [a&]:hover:text-accent-foreground",
      },
    },
    defaultVariants: {
      variant: "default",
    },
  }
)


function Badge({
  className,
  variant,
  asChild = false,
  ...props
}: React.ComponentProps<"span"> &
  VariantProps<typeof badgeVariants> & { asChild?: boolean }) {
  const Comp = asChild ? Slot : "span"


  return (
    <Comp
      data-slot="badge"
      className={cn(badgeVariants({ variant }), className)}
      {...props}
    />
  )
}


export { Badge, badgeVariants }

Alright so the component that the agent generated is quite a bit different than the shadcn version. Here are some of the differences that I notice on the agent version:

  • The destructive variant is missing from the Badge variants
  • The default variant styles are different:
    • Manually specifies colors rather than use Tailwind variables like primary or secondary
    • Missing the transition properties
    • Different padding values
    • Doesn't support anchor (<a/>) tags
  • Support for asChild is missing which is quite useful.
  • Exporting the badgeVariants is missing as well which is useful for types

Here's some additional reading on the asChild pattern and why it's important.

This is just the first component. I suspect that the others will be similar. I will note that shadcn components can be customized, but I asked the agent for just default shadcn components.

The manual color specifications that the agent did in these components got me curious... Maybe there isn't a Tailwind theme? And there's not. Whose fault would that be? Every frontend project that I've worked on has used a Tailwind theme. The theme is a core concept of Tailwind. This is a clear miss by the agent.

Conclusion

The agent definitely generated some code. No attempt was made at cleanup. What I asked for vs what I got was never the first attempt, but that's true for any engineering project. I'm disappointed in the agent not following shadcn or Tailwind standards. I've heard others say these agents operate intellectually at a junior level, and I think that's a fair comparison.

One final aspect of this process that is worth considering is time and cost. These agents are supposed to save us time right? Well I'm not convinced that any time was saved. I think this is something that I could have done in a couple hours. I still have to go through and verify all the questions and answers before sharing with my partner.
